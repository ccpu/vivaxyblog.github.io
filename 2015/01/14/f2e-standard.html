前端规范

14 Jan 2015

说明

前端规范是前端开发团队遵循和约定的代码书写规范。

规范为参考规范，不全是硬性要求，部分硬性约定见下一条书写规范。

提高代码的规范性和可维护性，让所有代码都是有规可循的，并且能够得到沉淀，减少重复劳动。

目录结构

.
├── index.html        入口页
├── js/               js
│   ├── lib/          js库
│   └── main.js       js入口
└── css/              css
    ├── lib/          css库
    ├── image/        图片
    └── main.css      主css

文件格式

  • 文本文件： UTF-8(无BOM)编码

html

  • css文件引用<link type="text/css" rel="stylesheet" href="...">放置于</head>前，不使用<style>

  • js文件引用<script type="text/javascript" src="..."></script>放置于</body>前，不使用<script>

  • css样式附着于class="..."，不使用style="..."

  • js操作句柄附着于class="j-xxx"

  • 绑定的数据使用data-xxx="..."

  • 文件头采用html5标准<!DOCTYPE html>

  • 特殊符号采用HTML符号实体

  • 标签一律由/关闭，如：<div></div>，<br />，<hr />，中间有空格

  • 采用双引号

js

  • 避免使用全局变量

  • 语句结束使用;

  • 采用单引号

  • 用变量形式申明函数

  • 变量在使用前必须通过 var 定义

css

  • 在可以使用缩写的情况下，尽量使用属性缩写，如：
/* good */
.post {
    font: 12px/1.5 arial, sans-serif;
}

/* bad */
.post {
    font-family: arial, sans-serif;
    font-size: 12px;
    line-height: 1.5;
}
  • 属性书写顺序：同一 rule set 下的属性在书写时，应按功能进行分组，并以 Formatting Model（布局方式、位置） > Box Model（尺寸） > Typographic（文本相关） > Visual（视觉效果） 的顺序书写，以提高代码的可读性。

Formatting Model 相关属性包括：position / top / right / bottom / left / float / display / overflow 等

Box Model 相关属性包括：border / margin / padding / width / height 等

Typographic 相关属性包括：font / line-height / text-align / word-wrap 等

Visual 相关属性包括：background / color / transition / list-style 等

另外，如果包含 content 属性，应放在最前面。

  • url() 函数中的路径不加引号，如：
body {
    background: url(bg.png);
}
  • 长度为 0 时须省略单位
/* good */
body {
    padding: 0 5px;
}

/* bad */
body {
    padding: 0px 5px;
}
  • 颜色值可以缩写时，必须使用缩写形式，如：
/* good */
.success {
    background-color: #aca;
}

/* bad */
.success {
    background-color: #aaccaa;
}

  • 颜色值不允许使用命名色值，如：black，green

  • 采用双引号

行内空格

  • 选择器 与 { 之间必须包含空格，如：
.selector {
}
  • 属性名 与之后的 : 之间不允许包含空格，如：
margin: 0;

  • 行结尾添加;

  • ,前没有空格，后面有空格，如：

font-family: Arial, sans-serif;

注释

  • /* css 注释 */

  • <!-- html 注释 -->

  • // js 单行注释

  • js 文档

/**
 * bla
 * foo
 * bar
 * @file 文件说明
 * @author 开发者姓名(开发者邮箱)
 * @since 新建时间
 * @modified 修改时间
 * @namespace 命名空间
 * @class //标记类
 * @extends 描述继承关系
 * @param {string} p1 参数1的说明
 * @return 返回值描述
 * @inner //标记内部函数，外部无法访问
 * @override //重写父类中的方法
 * @event 事件描述
 * @fires Select#change 广播事件
 * @const //标记常量
 * @type {string} //变量类型
 */

命名规范

  • 英文小写字母+数字或连接符-，用在文件夹，文件，样式名中，如：mengniu-m-20131230，dp-share.0.0.1.js，app-pop.0.0.7.min.js，top-box，box-list-item-1

  • 小写开头的驼峰式，用在js中表示变量，如：topBox，topBoxList，footerCopyright

  • 大写开头的驼峰式，用在js中表示构造器函数，如：SuperMBox

  • 方法内的私有变量加前缀_

需要表意明确

缩进

  • css，html，xml采用4个空格

  • js，json采用2个空格

外部链接中的协议可以省略

<script type="text/javascript" src="//www.google.com/some.js"></script>

background: url(//baidu.com/img/bg.png) no-repeat 0 0;

版本

  • 版本格式：主版本号.次版本号.修订号

  1. 主版本号：当你做了不兼容的API 修改，

  2. 次版本号：当你做了向下兼容的功能性新增，

  3. 修订号：当你做了向下兼容的问题修正。

先行版本号及版本编译信息可以加到“主版本号.次版本号.修订号”的后面，作为延伸。
